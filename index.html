1. Intro BD Relacional

Una base de datos relacional es un sistema que almacena información en tablas compuestas por filas y columnas.

¿Qué es?

Una base de datos relacional organiza datos en tablas que se relacionan mediante claves. Cada fila es un registro y cada columna un campo.

Concepto

Organiza la información en diferentes tablas conectadas entre sí. Cada tabla contiene registros (filas) y campos (columnas), lo que facilita el almacenamiento, búsqueda y gestión de datos.

Historia

En 1970, Edgar F. Codd, investigador de IBM, propuso el modelo relacional para organizar datos mediante tablas relacionadas. Este modelo revolucionó la forma de almacenar información y se convirtió en el estándar utilizado por empresas e instituciones de todo el mundo.

¿Qué es un SGBD?

Un Sistema Gestor de Bases de Datos (SGBD) es un software que permite crear, administrar, consultar y proteger bases de datos.

Funciones principales
  • Almacenar información.
  • Consultar datos rápidamente.
  • Modificar registros.
  • Controlar el acceso de los usuarios.
  • Mantener la seguridad de los datos.
Ejemplos de SGBD

MySQL, Oracle Database, SQL Server, PostgreSQL.

Importancia

Las bases de datos relacionales son fundamentales en escuelas, hospitales, bancos y aplicaciones web, ya que permiten gestionar grandes cantidades de información de forma segura, organizada y eficiente.

La imagen muestra cómo dos tablas se conectan mediante una clave foránea para mantener la consistencia entre clientes y pedidos. Esto refuerza la idea de que las bases de datos relacionales almacenan datos en estructuras enlazadas, no en archivos independientes.
-- Ejemplo de tablas relacionadas
CREATE TABLE Clientes (
  ID_Cliente INT PRIMARY KEY,
  Nombre VARCHAR(100),
  Correo VARCHAR(150)
);

CREATE TABLE Pedidos (
  ID_Pedido INT PRIMARY KEY,
  ID_Cliente INT,
  Producto VARCHAR(100),
  FOREIGN KEY (ID_Cliente) REFERENCES Clientes(ID_Cliente)
);
Ejemplos:
  • Un sistema bancario usa tablas para clientes, cuentas y transacciones relacionadas.
  • Una tienda online guarda productos y pedidos en tablas vinculadas.
  • Un hospital almacena pacientes, citas y médicos en tablas distintas con claves.
  • Una librería registra autores, libros y ventas sin duplicar información.
  • Una aplicación escolar consulta notas de alumnos junto a sus cursos.
Ejercicio: ¿Qué ventajas aporta un SGBD frente a guardar datos en archivos separados?
Permite relaciones entre tablas en lugar de archivos aislados. Ejemplos: usar JOIN para combinar clientes y pedidos, SELECT para filtrar datos, y constraints para mantener integridad. También ofrece seguridad, transacciones y menor redundancia.

2. Diseño de Base de Datos

Buenas prácticas: identificar entidades, atributos y relaciones; normalizar hasta 3NF cuando proceda.

El diseño de la base de datos define cómo se almacenan y se relacionan las tablas. Una estructura bien pensada evita redundancia y hace las consultas más rápidas y seguras.
El gráfico apoya el diseño mostrando cómo se identifican entidades, atributos y relaciones antes de crear las tablas. Es un recordatorio visual de que un diseño correcto evita redundancias y facilita las consultas posteriores.
-- Ejemplo: entidades Clientes y Pedidos
CREATE TABLE clientes (id INT PRIMARY KEY, nombre VARCHAR(100));
CREATE TABLE pedidos (id INT PRIMARY KEY, cliente_id INT, total DECIMAL(10,2),
  FOREIGN KEY (cliente_id) REFERENCES clientes(id));

-- Ejemplo adicional: vista de pedidos por cliente
SELECT c.nombre, p.id, p.total FROM clientes c
JOIN pedidos p ON c.id = p.cliente_id;
Ejemplos:
  • Entidades: cliente, pedido, producto.
  • Atributos: nombre, dirección, precio, fecha.
  • Relaciones: un cliente puede tener muchos pedidos.
  • Una tabla de detalle de pedido conecta pedidos y productos.
  • Una vista puede mostrar clientes con totales de ventas.
Ejercicio: Dibuja (mentalmente) el diagrama entidad-relación para clientes y pedidos e indica cardinalidades.
Un cliente puede tener 0..N pedidos; cada pedido pertenece a 1 cliente. Ejemplos: 1:N entre clientes y pedidos, 1:1 entre usuario y perfil, N:M entre estudiantes y cursos.

3. Crear BD y tablas / Restricciones

Crear bases de datos y tablas, y añadir constraints: PRIMARY KEY, UNIQUE, NOT NULL, CHECK.

Las restricciones aseguran la calidad de los datos: PRIMARY KEY identifica registros, UNIQUE evita duplicados, NOT NULL obliga un valor y CHECK verifica condiciones lógicas.
Estas imágenes destacan las reglas clave para crear bases de datos y tablas: definir correctamente tipos de datos, llaves y restricciones. Sirven como guía visual para estructurar una base de datos segura y eficiente.
CREATE DATABASE tienda;
USE tienda;
CREATE TABLE producto (
  id INT PRIMARY KEY,
  sku VARCHAR(50) UNIQUE,
  nombre VARCHAR(200) NOT NULL,
  precio DECIMAL(8,2) CHECK (precio >= 0)
);

-- Ejemplo adicional: restricción NOT NULL y bloqueo de precio negativo
CREATE TABLE proveedores (
  id INT PRIMARY KEY,
  nombre VARCHAR(120) NOT NULL,
  telefono VARCHAR(20) UNIQUE,
  monto_credito DECIMAL(10,2) CHECK (monto_credito >= 0)
);
Ejemplos:
  • PRIMARY KEY identifica de forma única cada fila.
  • UNIQUE evita duplicar códigos de producto o correos.
  • NOT NULL garantiza que un campo siempre tenga valor.
  • CHECK valida condiciones como precios mayores o iguales a cero.
  • Declarar constraints ayuda a mantener datos coherentes.
Ejercicio: Escribe la sentencia para crear una tabla `categorias` con `id` e `nombre` único.
CREATE TABLE categorias (id INT PRIMARY KEY, nombre VARCHAR(100) UNIQUE NOT NULL); O también: CREATE TABLE categorias (id INT NOT NULL PRIMARY KEY, nombre VARCHAR(100) NOT NULL, CONSTRAINT uq_nombre UNIQUE(nombre));

4. Modificar y eliminar Tablas

ALTER TABLE para cambiar estructura; DROP TABLE para eliminar.

Puedes ajustar una tabla existente sin perder datos usando ALTER TABLE. DROP TABLE borra la tabla completa, así que úsalo con precaución y preferiblemente con respaldo.
Las ilustraciones muestran los pasos para alterar una tabla y para eliminarla con cuidado. Refuerzan visualmente que alterar es modificar la estructura sin perder datos, mientras que eliminar borra la tabla completa.
-- Añadir columna
ALTER TABLE usuarios ADD COLUMN fecha_registro DATE;
-- Renombrar columna
ALTER TABLE clientes RENAME COLUMN nombre TO nombre_completo;
-- Eliminar tabla
DROP TABLE temporal;
Ejemplos:
  • ALTER TABLE usuarios ADD COLUMN telefono VARCHAR(20);
  • ALTER TABLE clientes RENAME COLUMN nombre TO nombre_completo;
  • ALTER TABLE productos DROP COLUMN descripcion;
  • ALTER TABLE pedidos MODIFY COLUMN total DECIMAL(12,2);
  • DROP TABLE temporal; elimina la tabla completa y sus datos.
Ejercicio: ¿Cómo renombrarías la columna `nombre` a `nombre_completo` en `clientes`?
ALTER TABLE clientes RENAME COLUMN nombre TO nombre_completo; (MySQL/Postgres) -- SQL Server: EXEC sp_rename 'clientes.nombre','nombre_completo','COLUMN';

5. Foreign Key / Normalización

Las claves foráneas mantienen relaciones y la normalización evita redundancias.

Una foreign key une tablas para que los datos sean consistentes. Normalizar divide la información en tablas relacionadas para evitar duplicados y errores al actualizar.
La imagen ilustra cómo una clave foránea conecta tablas y evita inconsistencias. Es un apoyo visual para entender que la normalización separa datos relacionados en tablas vinculadas.
CREATE TABLE ordenes (
  id INT PRIMARY KEY,
  cliente_id INT,
  fecha DATE,
  total DECIMAL(10,2),
  FOREIGN KEY (cliente_id) REFERENCES clientes(id)
);

-- Ejemplo adicional: clave foránea con eliminación en cascada
CREATE TABLE orden_detalle (
  id INT PRIMARY KEY,
  orden_id INT,
  producto_id INT,
  cantidad INT,
  FOREIGN KEY (orden_id) REFERENCES ordenes(id) ON DELETE CASCADE
);
Ejemplos:
  • Una tabla detalle usa foreign keys para vincular productos y pedidos.
  • Normalizar evita guardar el mismo cliente varias veces.
  • Una relación 1:N enlaza clientes con múltiples órdenes.
  • Separar direcciones en otra tabla reduce redundancias.
  • Actualizar un precio en una sola tabla evita inconsistencias.
Ejercicio: ¿Qué problema resuelve la normalización respecto a las actualizaciones de datos?
Evita anomalías de actualización y duplicación de datos. Ejemplos: no repetir datos de cliente en cada pedido, almacenar direcciones en una tabla separada, usar foreign key para mantener consistencia.

6. Insertar Registros

INSERT INTO agrega nuevos registros a una tabla. Si la columna `id` es AUTO_INCREMENT, el valor se genera automáticamente.

INSERT INTO es la forma básica de agregar datos. Primero se crea la base de datos y la tabla, luego se insertan registros y se comprueba con SELECT.
La imagen refleja el proceso de insertar nuevos registros después de crear la estructura de la tabla. Refuerza la idea de que primero se define la tabla y luego se agregan los datos con INSERT INTO.

CREATE DATABASE ejemplo_db;
USE ejemplo_db;

CREATE TABLE alumnos (
  id_alumno VARCHAR(10),
  nombre VARCHAR(50),
  apellido VARCHAR(50),
  edad INT,
  carrera VARCHAR(50)
);

INSERT INTO alumnos VALUES ('005','Luis','Ramirez',20,'Programacion');

-- Ver registros

Ejemplos:

7. Eliminar Registros

DELETE es la sentencia que se usa para borrar uno o más registros de una tabla según una condición.

Antes de eliminar registros, verifica qué filas se borrarán con SELECT. Usa WHERE siempre que no quieras eliminar todas las filas por accidente.
La imagen enfatiza la importancia de usar condiciones precisas al eliminar registros. Sirve como recordatorio visual de que DELETE sin WHERE puede borrar toda la tabla.
-- Eliminar un registro por ID
DELETE FROM usuarios
WHERE id = 4;

-- Eliminar por nombre
DELETE FROM usuarios
WHERE nombre = 'Carlos';

-- Eliminar varios registros según una condición
DELETE FROM usuarios
WHERE edad >= 22;

-- Eliminar con varias condiciones
DELETE FROM usuarios
WHERE ciudad = 'Puebla' AND edad = 30;

-- Eliminar con IN
DELETE FROM usuarios
WHERE id IN (2, 5, 8);
  • DELETE FROM: elimina registros.
  • WHERE: evita borrar todos los registros por accidente.
  • SELECT antes de DELETE: verifica qué datos se eliminarán.
  • DELETE FROM usuarios; borra todos los registros, pero la tabla sigue existiendo.
Ejemplos:
  • DELETE FROM usuarios WHERE id = 4;
  • DELETE FROM usuarios WHERE nombre = 'Carlos';
  • DELETE FROM usuarios WHERE edad >= 22;
  • DELETE FROM usuarios WHERE ciudad = 'Puebla' AND edad = 30;
  • DELETE FROM usuarios WHERE id IN (2, 5, 8);
Ejercicio: Elimina los usuarios cuyo `id` sea 2, 5 u 8.
DELETE FROM usuarios WHERE id IN (2, 5, 8); DELETE FROM usuarios WHERE id = 2 OR id = 5 OR id = 8;

8. Actualizar Registros

UPDATE modifica valores de registros existentes; SET especifica las columnas y nuevos valores, y WHERE limita el alcance.

La sintaxis básica es UPDATE [tabla] SET [columna] = 'valor' WHERE [condición]. Usa siempre WHERE, idealmente con la clave primaria (`id`) para actualizar un único registro.
La ilustración subraya cómo UPDATE cambia valores de registros existentes. Sirve para recordar que SET define las columnas a modificar y WHERE limita el alcance de la actualización.
-- Actualizar un registro
UPDATE usuarios
SET edad = 26
WHERE id = 1;

-- Actualizar varios campos
UPDATE usuarios
SET email = 'ana_nuevo@mail.com', edad = 27
WHERE id = 1;

-- Actualizar registros según condición
UPDATE usuarios
SET ciudad = 'Monterrey'
WHERE id > 1;

-- Ejemplo de formato de datos
UPDATE usuarios
SET nombre = 'Juan', fecha_registro = '2023-10-27'
WHERE id = 5;
Reglas de formato y tipos de datos
Tipo de Dato Usa comillas simples? Ejemplo
VARCHAR / TEXT Sí SET nombre = 'Mario'
DATE / DATETIME Sí SET fecha = '2023-10-27'
INT / DECIMAL No SET id = 5
  • UPDATE [tabla]: indica la tabla a modificar.
  • SET: define las columnas y sus nuevos valores.
  • WHERE: evita cambios masivos no deseados.
  • Sin WHERE, se actualizan todos los registros de la tabla.
Ejemplos:
  • UPDATE usuarios SET edad = 26 WHERE id = 1;
  • UPDATE usuarios SET email = 'ana_nuevo@mail.com', edad = 27 WHERE id = 1;
  • UPDATE usuarios SET ciudad = 'Monterrey' WHERE id > 1;
  • UPDATE usuarios SET nombre = 'Juan' WHERE id = 5;
  • UPDATE usuarios SET fecha_registro = '2023-10-27' WHERE id = 5;
Ejercicio: Actualiza el correo y la edad del usuario con id = 1.
UPDATE usuarios SET email = 'ana_nuevo@mail.com', edad = 27 WHERE id = 1; UPDATE usuarios SET email = 'ana_nuevo@mail.com' WHERE id = 1; UPDATE usuarios SET edad = 27 WHERE id = 1;

9. Seleccionar Registros

El comando SELECT se utiliza para consultar y mostrar información almacenada en una tabla.

SELECT es la base de todas las consultas. Permite ver todos los registros, elegir columnas específicas, filtrar con WHERE, ordenar resultados y obtener totales o valores únicos.
La imagen muestra la selección de datos con SELECT, reforzando que esta sentencia es la base para consultar la información guardada en las tablas. Es una ayuda visual para comprender consultas simples y filtradas.
-- 1. Seleccionar todos los registros
SELECT *
FROM Estudiantes;

-- 2. Seleccionar columnas específicas
SELECT nombre, edad
FROM Estudiantes;

-- 3. Filtrar registros con WHERE
SELECT *
FROM Estudiantes
WHERE carrera = 'Programación';

-- 4. Ordenar registros
SELECT *
FROM Estudiantes
ORDER BY nombre;

-- 5. Contar registros
SELECT COUNT(*)
FROM Estudiantes;

-- 6. Buscar por ID
SELECT *
FROM Estudiantes
WHERE id_estudiante = '001';

-- 7. Mostrar registros únicos
SELECT DISTINCT carrera
FROM Estudiantes;
Ventajas de SELECT
  • Consultar información rápidamente.
  • Filtrar datos específicos con WHERE.
  • Ordenar resultados con ORDER BY.
  • Contar registros con COUNT.
  • Analizar información almacenada.
Ejemplo final 
SELECT nombre, carrera
FROM Estudiantes
WHERE edad >= 18
ORDER BY nombre;

Obtiene el nombre y la carrera de los estudiantes mayores o iguales a 18 años, ordenados alfabéticamente por nombre.

Nota: los nombres de tablas y columnas pueden variar según la base de datos.

Ejemplos:
  • SELECT * FROM Estudiantes;
  • SELECT nombre, edad FROM Estudiantes;
  • SELECT * FROM Estudiantes WHERE carrera = 'Programación';
  • SELECT DISTINCT carrera FROM Estudiantes;
  • SELECT COUNT(*) FROM Estudiantes WHERE edad >= 18;
Ejercicio: Escribe una consulta que devuelva estudiantes de la carrera 'Redes' ordenados por apellido.
SELECT * FROM Estudiantes WHERE carrera = 'Redes' ORDER BY apellido; SELECT nombre, apellido FROM Estudiantes WHERE carrera = 'Redes' ORDER BY apellido;

10. Operadores lógicos y de comparación

Uso de AND, OR, NOT, LIKE, IS NULL, BETWEEN, IN para filtrar resultados.

Los operadores lógicos permiten combinar varias condiciones. LIKE busca patrones en texto, BETWEEN define rangos y IN compara con una lista de valores.
El gráfico complementa la sección mostrando cómo combinar condiciones con AND, OR, NOT y otros operadores. Refuerza la idea de que las consultas pueden ser más precisas y potentes mediante filtros lógicos.
SELECT * FROM productos WHERE precio BETWEEN 10 AND 50 AND nombre LIKE '%Pro%';
SELECT * FROM usuarios WHERE id IN (1,2,3) AND email IS NOT NULL;

-- Uso de NOT e IS NULL
SELECT * FROM clientes WHERE estado <> 'inactivo' AND email IS NOT NULL;
Ejemplos:
  • SELECT * FROM productos WHERE precio BETWEEN 10 AND 50;
  • SELECT * FROM usuarios WHERE id IN (1,2,3);
  • SELECT * FROM clientes WHERE email IS NOT NULL;
  • SELECT * FROM clientes WHERE estado <> 'inactivo';
  • SELECT * FROM productos WHERE nombre LIKE '%Pro%';
Ejercicio: Filtra órdenes del cliente 5 con total mayor a 100 o con estado 'urgente'.
SELECT * FROM ordenes WHERE cliente_id = 5 AND (total > 100 OR estado = 'urgente'); SELECT * FROM ordenes WHERE (cliente_id = 5 AND total > 100) OR (cliente_id = 5 AND estado = 'urgente');

11. Order By / Limit / Top

Ordenar resultados y limitar filas devueltas.

ORDER BY organiza los resultados por una o más columnas. LIMIT o TOP reducen la cantidad de filas devueltas, lo cual es útil para paginación o vistas rápidas.
Estas imágenes muestran cómo ordenar resultados y limitar filas. Refuerzan visualmente la diferencia entre ORDER BY y LIMIT/TOP para extraer subconjuntos de datos ordenados.
SELECT * FROM productos ORDER BY precio DESC LIMIT 10; -- MySQL/Postgres
-- SQL Server: SELECT TOP 10 * FROM productos ORDER BY precio DESC;

-- Paginación simple
SELECT * FROM productos ORDER BY nombre ASC LIMIT 10 OFFSET 20;
Ejemplos:
  • SELECT * FROM productos ORDER BY precio DESC LIMIT 10;
  • SELECT TOP 5 * FROM ventas ORDER BY fecha DESC;
  • SELECT * FROM empleados ORDER BY apellido ASC LIMIT 20;
  • SELECT * FROM clientes ORDER BY nombre ASC LIMIT 15 OFFSET 5;
  • SELECT * FROM pedidos ORDER BY total DESC LIMIT 3;
Ejercicio: Devuelve los 5 clientes con más pedidos.
SELECT c.id, c.nombre, COUNT(o.id) AS num_pedidos FROM clientes c JOIN ordenes o ON c.id = o.cliente_id GROUP BY c.id, c.nombre ORDER BY num_pedidos DESC LIMIT 5; -- SQL Server SELECT TOP 5 c.id, c.nombre, COUNT(o.id) AS num_pedidos FROM clientes c JOIN ordenes o ON c.id = o.cliente_id GROUP BY c.id, c.nombre ORDER BY num_pedidos DESC;

12. Funciones de texto

Funciones como CONCAT, SUBSTRING, UPPER, LOWER, TRIM, LENGTH.

Las funciones de texto permiten transformar y evaluar cadenas. Útiles para normalizar datos, extraer partes de un email o construir informes personalizados.
SELECT CONCAT(nombre,' ',apellido) AS nombre_completo FROM usuarios;
SELECT UPPER(email) FROM usuarios;

-- Más ejemplos de texto
SELECT TRIM(nombre) AS limpio, LENGTH(email) AS largo_email FROM usuarios;
SELECT SUBSTRING(nombre,1,10) AS prefijo FROM usuarios;
SELECT REPLACE(nombre,'Maria','María') AS nombre_corregido FROM usuarios;
SELECT CONCAT(SUBSTRING(nombre,1,1),'. ',apellido) AS iniciales FROM usuarios;

13. Funciones numéricas

SUM, AVG, COUNT, MIN, MAX, ROUND, FLOOR, CEIL.

Las funciones numéricas permiten calcular totales, promedios y estadísticas. Son clave para reportes financieros y análisis de datos en SQL.
La imagen apoya la sección de funciones numéricas mostrando ejemplos de cálculos y agregaciones. Sirve para recordar que SQL puede resumir grandes cantidades de datos con SUM, AVG, COUNT, MIN y MAX.
SELECT AVG(total) AS promedio FROM ordenes;
SELECT COUNT(*) FROM usuarios WHERE fecha_registro > '2024-01-01';

-- Ejemplo adicional: sumas, mínimos y máximos
SELECT SUM(total) AS total_ventas, MIN(total) AS venta_minima, MAX(total) AS venta_maxima FROM ordenes;
SELECT ROUND(AVG(total), 2) AS promedio_redondeado FROM ordenes;
SELECT FLOOR(precio) AS precio_redondeado, CEIL(precio) AS precio_techo FROM productos;
SELECT SUM(cantidad) AS total_items FROM orden_detalle;
Funciones numéricas clave
Función Concepto Ejemplo SQL Qué obtiene
SUM Suma todos los valores de una columna. SELECT SUM(precio) FROM productos; Suma el costo de todos los productos en inventario.
AVG Calcula el promedio aritmético de los valores. SELECT AVG(precio) FROM productos; Obtiene el precio promedio de los productos.
COUNT Cuenta la cantidad de registros o filas. SELECT COUNT(*) FROM productos; Indica cuántos productos hay registrados.
MAX Encuentra el valor más alto en la columna. SELECT MAX(precio) FROM productos; Muestra el precio del producto más caro.
MIN Encuentra el valor más bajo en la columna. SELECT MIN(precio) FROM productos; Muestra el precio del producto más barato.
Ejemplos:
  • SELECT SUM(total) AS total_ventas FROM ordenes;
  • SELECT AVG(total) AS promedio FROM ordenes;
  • SELECT COUNT(*) FROM usuarios WHERE fecha_registro > '2024-01-01';
  • SELECT MAX(precio) FROM productos;
  • SELECT MIN(precio) FROM productos;
  • SELECT FLOOR(precio) AS precio_redondeado FROM productos;
  • SELECT CEIL(precio) AS precio_techo FROM productos;
Ejercicio: Calcula la suma total de ventas por mes.
SELECT DATE_TRUNC('month', fecha) AS mes, SUM(total) FROM ordenes GROUP BY mes ORDER BY mes; En MySQL: SELECT DATE_FORMAT(fecha,'%Y-%m') AS mes, SUM(total) FROM ordenes GROUP BY mes ORDER BY mes; En SQL Server: SELECT FORMAT(fecha,'yyyy-MM') AS mes, SUM(total) FROM ordenes GROUP BY FORMAT(fecha,'yyyy-MM') ORDER BY mes;

14. Manejo de Fechas y horas

Funciones: NOW(), DATE_ADD/DATE_SUB, DATE_TRUNC, EXTRACT.

Las fechas y horas se tratan con funciones especiales. Sirven para filtrar intervalos, agrupar por periodos y hacer cálculos temporales precisos.
Las imágenes refuerzan el uso de funciones de fecha para filtrar y agrupar datos temporales. Ayudan a visualizar cómo trabajar con intervalos, extracciones de año y agregaciones por día.
-- Últimos 7 días

15. JOIN básico

Combinar tablas con JOIN para obtener filas relacionadas de más de una tabla.

INNER JOIN, LEFT JOIN y RIGHT JOIN permiten unir datos de tablas distintas según una condición de relación.
SELECT c.nombre, o.total
FROM clientes c
JOIN ordenes o ON c.id = o.cliente_id;

-- LEFT JOIN para ver todos los clientes aunque no tengan órdenes
SELECT c.nombre, o.total
FROM clientes c
LEFT JOIN ordenes o ON c.id = o.cliente_id;

-- Combinar varias tablas con alias
SELECT c.nombre, o.id AS pedido_id, p.nombre AS producto
FROM clientes c
JOIN ordenes o ON c.id = o.cliente_id
JOIN productos p ON o.producto_id = p.id;
Ejemplos:
  • SELECT c.nombre, o.total FROM clientes c JOIN ordenes o ON c.id = o.cliente_id;
  • SELECT c.nombre, o.total FROM clientes c LEFT JOIN ordenes o ON c.id = o.cliente_id;
  • SELECT c.nombre, o.total FROM clientes c RIGHT JOIN ordenes o ON c.id = o.cliente_id;
  • SELECT c.nombre, p.nombre FROM clientes c JOIN ordenes o ON c.id = o.cliente_id JOIN productos p ON o.producto_id = p.id;
Ejercicio: Muestra nombre de cliente y total de orden usando JOIN.
SELECT c.nombre, o.total FROM clientes c JOIN ordenes o ON c.id = o.cliente_id; También: SELECT clientes.nombre, ordenes.total FROM clientes JOIN ordenes ON clientes.id = ordenes.cliente_id;

16. Combinaciones internas y externas

INNER JOIN devuelve solo coincidencias; LEFT/RIGHT JOIN devuelven todas las filas de una tabla y las coincidencias de la otra.

Las combinaciones internas y externas permiten ver datos relacionados y también filas sin coincidencia en una de las tablas.
-- INNER JOIN devuelve solo coincidencias
SELECT p.nombre, c.nombre AS cliente
FROM pedidos p
INNER JOIN clientes c ON p.cliente_id = c.id;

-- LEFT JOIN muestra todos los pedidos y clientes cuando existen
SELECT p.id, c.nombre
FROM pedidos p
LEFT JOIN clientes c ON p.cliente_id = c.id;

-- RIGHT JOIN muestra todos los clientes y pedidos coincidentes
SELECT p.id, c.nombre
FROM pedidos p
RIGHT JOIN clientes c ON p.cliente_id = c.id;

-- JOIN con alias y condiciones adicionales
SELECT p.id, c.nombre, o.total
FROM pedidos p
INNER JOIN clientes c ON p.cliente_id = c.id
INNER JOIN ordenes o ON o.cliente_id = c.id
WHERE o.total > 100;
Ejemplos:
  • INNER JOIN: solo registros que tienen relación en ambas tablas.
  • LEFT JOIN: todas las filas de la tabla izquierda y coincidencias de la derecha.
  • RIGHT JOIN: todas las filas de la tabla derecha y coincidencias de la izquierda.
  • Usar alias facilita consultas con varias tablas.
Ejercicio: Usa LEFT JOIN para mostrar clientes aunque no tengan pedidos.
SELECT c.nombre, o.id AS pedido_id FROM clientes c LEFT JOIN ordenes o ON c.id = o.cliente_id; También: SELECT clientes.nombre, ordenes.id AS pedido_id FROM clientes LEFT JOIN ordenes ON clientes.id = ordenes.cliente_id;

17. Subconsultas

Una subconsulta anidada devuelve datos que luego se usan en la consulta externa.

Las subconsultas permiten filtrar, comparar o calcular valores entre diferentes consultas sin usar tablas temporales.
-- Subconsulta en WHERE
SELECT nombre FROM clientes
WHERE id IN (SELECT cliente_id FROM ordenes WHERE total > 100);

-- Subconsulta en SELECT
SELECT nombre,
  (SELECT COUNT(*) FROM ordenes o WHERE o.cliente_id = c.id) AS total_pedidos
FROM clientes c;

-- Subconsulta con comparación de fecha
SELECT nombre FROM clientes
WHERE id = (SELECT cliente_id FROM ordenes WHERE fecha = '2024-01-01');
Ejemplos:
  • Buscar clientes con pedidos mayores a 100.
  • Contar órdenes de cada cliente dentro de una subconsulta.
  • Usar subconsulta para comparar un valor con un promedio.
  • Filtrar con una subconsulta que devuelve un solo valor.
Ejercicio: Selecciona clientes con más de 5 pedidos usando subconsulta.
SELECT nombre FROM clientes WHERE id IN (SELECT cliente_id FROM ordenes GROUP BY cliente_id HAVING COUNT(*) > 5); También: SELECT nombre FROM clientes c WHERE EXISTS (SELECT 1 FROM ordenes o WHERE o.cliente_id = c.id GROUP BY o.cliente_id HAVING COUNT(*) > 5);

18. Creación de vistas

CREATE VIEW guarda una consulta como una vista reutilizable.

Las vistas simplifican consultas complejas y mejoran la legibilidad al tratar resultados predefinidos como tablas virtuales.
CREATE VIEW vista_clientes_pedidos AS
SELECT c.nombre, o.id AS pedido_id, o.total
FROM clientes c
JOIN ordenes o ON c.id = o.cliente_id;

SELECT * FROM vista_clientes_pedidos;

CREATE VIEW vista_totales AS
SELECT cliente_id, SUM(total) AS total_ventas
FROM ordenes
GROUP BY cliente_id;

SELECT * FROM vista_totales WHERE total_ventas > 500;
Ejemplos:
  • Crear vista de clientes con totales de pedidos.
  • Usar la vista para consultar sin repetir la JOIN.
  • Las vistas facilitan informes y consultas compartidas.
  • Filtrar desde la vista como si fuera una tabla normal.
Ejercicio: Crea una vista para ver el nombre del cliente y el total de cada orden.
CREATE VIEW vista_ordenes AS SELECT c.nombre, o.total FROM clientes c JOIN ordenes o ON c.id = o.cliente_id; Luego usar: SELECT * FROM vista_ordenes; Alternativa: CREATE VIEW ordenes_cliente AS SELECT clientes.nombre, ordenes.total FROM clientes JOIN ordenes ON clientes.id = ordenes.cliente_id;

19. Agrupación de resultados / Group By / Having

GROUP BY agrupa filas; HAVING filtra grupos después de aplicar agregaciones.

GROUP BY resume datos por categoría, como totales por cliente. HAVING selecciona los grupos que cumplen condiciones específicas.
SELECT cliente_id, COUNT(*) AS pedidos, SUM(total) AS total_ventas
FROM ordenes
GROUP BY cliente_id
HAVING SUM(total) > 1000;

SELECT producto_id, AVG(precio) AS precio_promedio
FROM orden_detalle
GROUP BY producto_id
HAVING AVG(precio) > 100;
Ejemplos:
  • Contar pedidos por cliente.
  • Sumar ventas por día o por producto.
  • Filtrar grupos con HAVING.
  • Usar GROUP BY con varias columnas.
Ejercicio: Muestra clientes con más de 5 pedidos y el total de ventas.
SELECT cliente_id, COUNT(*) AS pedidos, SUM(total) AS total_ventas FROM ordenes GROUP BY cliente_id HAVING COUNT(*) > 5; También: SELECT cliente_id, SUM(total) AS total_ventas FROM ordenes GROUP BY cliente_id HAVING SUM(total) > 1000;

20. Full outer join / Cross join

FULL OUTER JOIN combina todas las filas de ambas tablas; CROSS JOIN genera el producto cartesiano.

FULL OUTER JOIN devuelve filas coincidentes y no coincidentes de ambas tablas. CROSS JOIN combina cada fila de una tabla con todas las filas de la otra.
-- Full outer join (puede variar según SGBD)
SELECT c.nombre, o.id AS pedido_id
FROM clientes c
FULL OUTER JOIN ordenes o ON c.id = o.cliente_id;

-- Cross join: cada cliente con cada producto
SELECT c.nombre, p.nombre AS producto
FROM clientes c
CROSS JOIN productos p;

-- Producto cartesiano con filtro
SELECT c.nombre, p.nombre AS producto
FROM clientes c
CROSS JOIN productos p
WHERE p.precio > 50;
  • FULL OUTER JOIN devuelve todas las filas, incluso sin coincidencia.
  • CROSS JOIN produce todas las combinaciones posibles.
  • Usa CROSS JOIN con cuidado: puede generar muchas filas.
  • Filtrar los resultados reduce el volumen del producto cartesiano.
Ejercicio: Crea una consulta CROSS JOIN que combine clientes y productos.
SELECT c.nombre, p.nombre FROM clientes c CROSS JOIN productos p; También: SELECT clientes.nombre, productos.nombre FROM clientes CROSS JOIN productos;

21. Procedimientos almacenados

Un procedimiento almacenado encapsula lógica SQL y se guarda en el servidor.

Los procedimientos guardan consultas, cálculos y actualizaciones en una rutina que se puede ejecutar con parámetros.
CREATE PROCEDURE ObtenerPedidosCliente(IN clienteID INT)
BEGIN
  SELECT * FROM ordenes WHERE cliente_id = clienteID;
END;

CALL ObtenerPedidosCliente(3);

CREATE PROCEDURE AgregarCliente(IN p_nombre VARCHAR(100))
BEGIN
  INSERT INTO clientes(nombre) VALUES (p_nombre);
END;

CALL AgregarCliente('Luis');
Beneficios:
  • Reutilización de lógica.
  • Menos errores al repetir consultas.
  • Ejecuta tareas complejas en el servidor.
  • Puedes recibir parámetros y devolver resultados.
Ejercicio: Define un procedimiento que devuelva órdenes de un cliente dado.
CREATE PROCEDURE ObtenerPedidosCliente(IN clienteID INT) BEGIN SELECT * FROM ordenes WHERE cliente_id = clienteID; END; CALL ObtenerPedidosCliente(3); También: CREATE PROCEDURE ObtenerPedidosCliente (IN p_cliente_id INT) BEGIN SELECT id, total, fecha FROM ordenes WHERE cliente_id = p_cliente_id; END;

22. Triggers

Triggers ejecutan acciones automáticamente cuando ocurren eventos como INSERT, UPDATE o DELETE.

Los triggers permiten ejecutar validaciones o actualizaciones automáticas en la base de datos después de cambios en las tablas.
CREATE TRIGGER actualiza_fecha_modificacion
BEFORE UPDATE ON productos
FOR EACH ROW
SET NEW.fecha_modificacion = NOW();

CREATE TRIGGER historial_orden
AFTER INSERT ON ordenes
FOR EACH ROW
INSERT INTO historial_ordenes (orden_id, fecha) VALUES (NEW.id, NOW());
Ejemplos:
  • Registrar cambios de datos.
  • Actualizar fechas automáticamente.
  • Validar datos antes de insertarlos.
  • Guardar histórico de acciones automáticamente.
Ejercicio: Crea un trigger que actualice la fecha de modificación al actualizar un producto.
CREATE TRIGGER actualiza_fecha_modificacion BEFORE UPDATE ON productos FOR EACH ROW SET NEW.fecha_modificacion = NOW(); También: CREATE TRIGGER actualiza_fecha_modificacion AFTER UPDATE ON productos FOR EACH ROW BEGIN UPDATE productos SET fecha_modificacion = NOW() WHERE id = NEW.id; END;

23. Índices

Los índices aceleran las búsquedas y las consultas en columnas usadas frecuentemente en WHERE y JOIN.

Un índice aparece como una estructura interna que permite localizar filas más rápido sin escanear toda la tabla.
CREATE INDEX idx_cliente_id ON ordenes(cliente_id);
CREATE INDEX idx_email ON usuarios(email);
CREATE UNIQUE INDEX idx_orden_unica ON ordenes(orden_numero);

-- Ver índices en SQL Server
-- SELECT * FROM sys.indexes WHERE object_id = OBJECT_ID('ordenes');
Ventajas:
  • Consultas más rápidas.
  • Mejora el rendimiento en JOINs y filtros.
  • Útil en columnas de búsqueda frecuente.
  • Un índice único evita duplicados en valores críticos.
Ejercicio: Crea un índice para la columna `email` en `usuarios`.
CREATE INDEX idx_email ON usuarios(email); También: CREATE UNIQUE INDEX idx_email_unico ON usuarios(email);

24. UDF

Una UDF es una función definida por el usuario para reutilizar lógica dentro de consultas.

Las funciones UDF pueden devolver valores calculados y usarse en SELECT, WHERE o HAVING.
CREATE FUNCTION calcular_descuento(total DECIMAL(10,2))
RETURNS DECIMAL(10,2)
DETERMINISTIC
RETURN total * 0.9;

SELECT nombre, calcular_descuento(total) AS total_con_descuento FROM ordenes;

CREATE FUNCTION es_mayor_edad(edad INT)
RETURNS BOOLEAN
DETERMINISTIC
RETURN edad >= 18;

SELECT nombre FROM alumnos WHERE es_mayor_edad(edad);
Ejemplos:
  • Calcular descuentos o impuestos.
  • Normalizar cálculos complejos en una función.
  • Reutilizar lógica en varias consultas.
  • Usar funciones en WHERE, SELECT o HAVING.
Ejercicio: Crea una función que aplique un descuento del 10% al total.
CREATE FUNCTION calcular_descuento(total DECIMAL(10,2)) RETURNS DECIMAL(10,2) DETERMINISTIC RETURN total * 0.9; SELECT calcular_descuento(100) AS descuento; También: CREATE FUNCTION calcular_descuento(total DECIMAL(10,2)) RETURNS DECIMAL(10,2) BEGIN RETURN total * 0.9; END;
